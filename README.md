### Задание 1

` 1. CPU Usage (процент загрузки, load average)
Вычисления сильно нагружают CPU, нужно отслеживать приближение к 100% и деградацию производительности.
2. HTTP Requests (RPS, latency p50/p95/p99, error rate 4xx+5xx)
Основной протокол взаимодействия. Метрики показывают доступность сервиса, скорость ответов и количество ошибок.
3. Disk Usage (свободное место на диске)
Отчёты сохраняются на диск, что может привести к заполнению и падению сервиса.
4. Memory Usage (RSS/Heap + OOM)
При CPU-интенсивных задачах память часто становится следующим bottleneck.
5. Active Computations + Queue Length (если есть)
Позволяет видеть реальную нагрузку на бизнес-логику и вовремя выявлять заторы. `

### Задание 2

`1. CPUla (Load Average) — насколько сильно загружен процессор в среднем за 1, 5 и 15 минут.
RAM — объём оперативной памяти, которую использует сервис.
Inodes — количество файлов, которые может хранить диск (даже если место есть, inodes могут закончиться).

2. Метрики качества обслуживания клиентов (SLO):

Success Rate — процент успешно выполненных вычислений и выданных отчётов.
Availability (Uptime) — процент времени, когда сервис доступен (например, 99.9%).
Average / p95 Latency — среднее и максимальное время выполнения запроса.
SLA Compliance — % запросов, которые выполнены в пределах оговорённого времени.
Error Rate (с точки зрения клиента) — сколько запросов заканчиваются ошибкой.`
