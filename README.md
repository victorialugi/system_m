### Задание 2

`1. CPUla (Load Average) — насколько сильно загружен процессор в среднем за 1, 5 и 15 минут.
RAM — объём оперативной памяти, которую использует сервис.
Inodes — количество файлов, которые может хранить диск (даже если место есть, inodes могут закончиться).

2. Метрики качества обслуживания клиентов (SLO):

Success Rate — процент успешно выполненных вычислений и выданных отчётов.
Availability (Uptime) — процент времени, когда сервис доступен (например, 99.9%).
Average / p95 Latency — среднее и максимальное время выполнения запроса.
SLA Compliance — % запросов, которые выполнены в пределах оговорённого времени.
Error Rate (с точки зрения клиента) — сколько запросов заканчиваются ошибкой.`

### Задание 5

`1. Pull-модель (Prometheus, Zabbix и т.д.)
Сервер сам периодически забирает метрики.

Плюсы:

Легко контролировать частоту сбора
Хорошая предсказуемость нагрузки
Удобно для динамических сред (Kubernetes)

Минусы:

Сервер должен «знать» все цели (service discovery)
Проблемы с firewall (нужны входящие соединения)
Задержка при падении агента

Push-модель (InfluxDB, StatsD, Graphite, Prometheus Pushgateway)
Агенты сами отправляют метрики на сервер.

Плюсы:

Проще проходить через firewall (только исходящие соединения)
Быстрее доставка метрик
Легче работать с краткоживущими jobs

Минусы:

Сервер может быть заспамлен (нужна rate limiting)
Сложнее контролировать, кто и сколько шлёт
Потеря метрик при проблемах с сетью`
